📊 Profile 对比测试

验证不同基金画像是否对 Step10 Fit 判断产生预期差异

🤔 什么是 Profile（投资人画像）

Profile 是系统判断"项目适不适合"的上下文变量。同一个 BP，切换不同 Profile → Step10 Fit 判断不同。

🔵 neutral_investor

中性投资人（默认）

无硬约束，纯项目质量判断

不因落地/反投直接 pass

🟡 government_fund

政府产业基金

硬约束：落地、反投、税收贡献

不满足 → not_fit + pass

🔷 vc_fund

市场化VC

硬约束：增长性、退出路径

偏好：高增长赛道

🟣 industrial_fund

产业资本

硬约束：战略协同

偏好：上下游整合

🔑 核心差异：neutral vs government_fund

以"杉海创新科技"为例（该 BP 不承诺落地、反投条件苛刻）：

维度 🔵 neutral_investor（默认） 🟡 government_fund（政府基金）
Step0 硬约束 无硬约束 必须有落地意愿、反投要求、税收贡献承诺
Step5 会前判断 基于项目本身质量判断 可能因落地问题降低初步评分
Step7 Profile问题 无 profile 专属问题 验证落地意愿、反投能力等
Step10 fit_decision partial_fit not_fit
Step10 recommendation request_materials pass
关键触发点 不因落地/反投直接拒绝 落地意愿缺失 → 硬约束冲突 → 直接 pass
⚠️ 重要原则：Step6 永远不受 Profile 影响！会议记录的事实提取是纯客观的，不能被基金偏好污染。

🧪 对比测试步骤（按顺序做）

1

上传同一个 BP（建议用杉海创新科技）

新建两个项目，分别选择不同 Profile：

  • 项目 A → 选择 neutral_investor
  • 项目 B → 选择 government_fund

两个项目使用完全相同的 BP 文件，这样才能准确对比 Profile 影响。

创建新项目 →
2

运行 1.0 会前分析（Step1→5）

分别对两个项目运行 1.0 流程。完成后：

  • 打开项目 A 的结果页 → 记住 Step5 的会前判断
  • 打开项目 B 的结果页 → 对比 Step5 的会前判断是否不同

观察点：government_fund 版本是否对"落地意愿"等问题更敏感？

3

上传同一份模拟会议记录

两个项目使用相同的模拟会议记录（或粘贴同一段文本）。

推荐使用已有的测试会议记录： workspace/测试科技公司_20260424_173519/meeting/meeting.txt

Step6 的输出在两个项目中应该完全相同（不受 Profile 影响）。

4

运行 2.0 全流程（Step6→10）

两个项目都跑完 2.0 后，分别查看结果页。

观察点 1（Step7）：government_fund 版本是否有额外的 Profile 问题被验证？

观察点 2（Step10）：这是最关键的区别：

  • 项目 A（neutral）：应该是 partial_fit + request_materials
  • 项目 B（gov）：应该是 not_fit + pass（因为落地/反投硬约束冲突）
5

点击"📦 原始 JSON" tab 对比

在两个项目的 2.0 结果页 → 点击「📦 原始 JSON」tab，下载并对比：

  • step10.json — 重点对比 fit_decision、final_recommendation、matched_constraints、mismatched_constraints
  • step7.json — 对比 question_source 分布（neutral 无 profile 问题，gov 有）
  • step6.json — 确认两份 step6 完全相同（Profile 不影响事实提取）

✅ 验收清单（全部通过才算成功）

  • ✅ 项目 B（government_fund）Step10 的 fit_decision = not_fit（不是 partial_fit）
  • ✅ 项目 B（government_fund）Step10 的 final_recommendation = pass（不是 continue）
  • ✅ 项目 A（neutral_investor）Step10 的 final_recommendation = request_materials 或 continue（不是直接 pass）
  • ✅ 项目 B Step10 的 mismatched_constraints 包含"落地意愿"或"反投要求"
  • ✅ 两个项目的 step6.json 内容完全一致（Profile 不污染事实提取）
  • ✅ Step7 在 government_fund 版本中有 question_source=profile 的问题
  • ❌ 如果 government_fund 版本的 Step6 内容与 neutral 不同 → 说明 Profile 污染了 Step6，这是严重 bug！

🎯 测试通过的标准

同一个 BP + 同一个会议记录 → neutral_investor 和 government_fund 版本：

  • Step6（事实提取）：✅ 完全相同
  • Step7（问题验证）：⚠️ government 版本多出 profile 专属问题
  • Step10（Fit 判断）：❌ 明显不同（gov 直接 pass，neutral 继续 request_materials）

如果 Step10 结果相同，说明 Profile 硬约束没有生效，或 Step10 的规则引擎有问题。

⚠️ 当前已知限制

  • Step4 尚无 profile 区分标注：Step4 开会问题目前全部为 base 类型，profile 相关问题暂未在 step4 层面注入。这是后续迭代点。
  • Step10 依赖 Step9 结果：如果 Step9 项目层判断为 stop，Step10 无论如何都会是 not_fit。
  • Profile 切换后需重新跑流程：切换 Profile 只影响 step0.json，不会自动清空后续步骤的缓存。
